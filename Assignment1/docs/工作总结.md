# 白盒测试用例生成工具 — 工作总结

## 一、项目概述

本项目实现了一个基于 **LLM 大语言模型** 的白盒测试用例自动生成工具。工具以 Python CLI 形式运行，支持 Python 和 JavaScript 两种被测语言，支持多个 LLM 后端（DeepSeek Chat、DeepSeek Reasoner、Google Gemini、GitHub Models），具备覆盖率闭环反馈能力。

---

## 二、开发工作时间线

### 第1阶段：基础架构搭建
- 搭建项目目录结构（7 个模块）
- 实现 `PythonAnalyzer`（AST 代码分析器），提取函数列表、分支点、复合条件、执行路径
- 实现 `LLMClient`（统一 LLM 客户端），支持 OpenAI 兼容格式 + Google GenAI
- 实现 `prompts.py`（Prompt 模板），包含初始生成 prompt 和补充生成 prompt
- 实现 `TestGenerator`（主流程），串联 分析→提示→调用LLM→解析→生成测试文件
- 实现 `CoverageRunner`（覆盖率测量），支持 pytest + coverage.py（Python）和 Jest + istanbul（JavaScript）
- 实现 `ExperimentTracker`（实验记录），记录每次运行的数据并支持对比表格
- 实现 `main.py`（CLI 入口），支持交互模式和命令行参数模式

### 第2阶段：闭环反馈机制
- 实现覆盖率闭环：生成测试 → 运行覆盖率 → 未达标则反馈未覆盖行/分支给 LLM → 补充测试用例 → 再次运行
- 设置覆盖率目标（语句 90%、分支 80%）和最大轮次限制（默认 3 轮）
- 实现 JSON 解析容错：LLM 返回的 JSON 常有格式问题，实现了三级解析策略（直接解析 → 提取代码块 → 逐个对象提取）

### 第3阶段：创新点实现

#### 创新点1：多 LLM 对比实验（`--compare`）
- 实现 `--compare` 参数，自动使用 config.yaml 中所有 LLM 后端依次运行
- 生成对比表格和 Markdown 报告
- 在 Python 和 JavaScript 两个被测项目上进行了多组对比实验

#### 创新点2：AST 代码结构分析辅助（`--ast`）
- 实现 `--ast` 参数，利用 Python AST 模块精确提取代码结构信息注入 prompt
- 新增含 AST 信息的 prompt 模板（`build_ast_prompt()`）
- 实验验证：AST 辅助后闭环轮次从 3 轮降至 1 轮，用例准确率提升

#### 创新点3：自适应输出模式 — LLM 直接生成测试脚本（`--script`）
- **问题发现**：JSON 格式的测试用例无法表达 JavaScript 链式调用（如 `dayjs('2023-01-01').format('YYYY')`），导致 dayjs 覆盖率仅 ~30%
- **解决方案**：`--script` 模式下，LLM 先生成 JSON 用例，再用第二个 prompt 让 LLM 直接生成完整可运行的测试脚本
- 新增 `build_script_prompt()`：让 LLM 根据 JSON 用例 + 源代码生成 pytest/Jest 测试脚本
- 新增 `build_supplement_script_prompt()`：闭环补充时只生成测试方法片段（不含 import/describe 外壳），自动拼接到已有脚本
- 实现语法校验：`node --check`（JS）/ `ast.parse`（Python），防止有语法错误的脚本覆盖好的版本
- 实现回退机制：记录最佳覆盖率版本，新脚本覆盖率下降时自动回退
- 移除了 `--no-coverage` 参数，覆盖率闭环自动跟随 `--script` 模式

## 三、遇到的问题与解决方案

### 问题1：LLM 生成的脚本被截断导致覆盖率归零
**现象**：闭环补充时让 LLM 重新生成整个脚本（82个用例 → 366行），用例越多越容易被截断（`SyntaxError: Unexpected end of input`），Jest 无法运行，覆盖率变 0%。

**解决**：改为**片段拼接模式**——闭环补充时只让 LLM 生成补充的 test 方法片段（不含 import/describe 外壳），由代码自动拼接到已有脚本的 `});` 前。同时 prompt 只发送未覆盖行及上下文（前后3行），大幅减少 prompt 长度。

### 问题2：覆盖率回退
**现象**：第2轮覆盖率 86%，但第3轮 LLM 生成的脚本有错误导致覆盖率反而下降甚至归零。

**解决**：实现**回退机制**——记录每轮的 `best_code`、`best_cov`、`best_report`，如果新脚本覆盖率低于已有最佳值，自动回退到最佳版本。最终输出保证使用覆盖率最高的版本。

### 问题3：补充片段语法错误
**现象**：LLM 生成的测试方法片段虽然通过了基本检查，但拼接后可能存在括号不匹配等问题。

**解决**：拼接后使用 `node --check`（JS）或 `ast.parse`（Python）进行完整语法校验，语法错误则丢弃本次补充，保留上一轮的版本。

### 问题4：JavaScript 项目覆盖率低
**现象**：dayjs 是链式调用库，JSON 格式的测试用例只能生成 `index(input)` 形式的构造函数调用，无法触发内部方法，覆盖率仅 ~30%。

**解决**：实现 `--script` 模式，让 LLM 直接生成可运行的 Jest 测试脚本，支持链式调用写法，覆盖率提升至 64%~97%。

### 问题5：闭环补充准确率下降
**现象**：闭环补充的测试用例可能过度拟合未覆盖行，生成的断言逻辑不正确，导致准确率（通过率）下降。

**分析**：LLM 在闭环补充时更倾向于覆盖未覆盖行，但不一定符合正确的输入输出关系。这是 LLM 生成测试的固有局限性，后续可通过优化 prompt 或引入验证机制改善。

### 4.3 关键结论

1. **AST 分析的效果**：使用 `--ast` 后，Python 项目闭环轮次从 3 轮降至 1 轮（首轮即达标），用例准确率显著提升（如 deepseek-chat 从 18/8 提升到 17/1）。
2. **`--script` 模式的效果**：JavaScript 项目覆盖率从 ~30% 提升到 64%~97%，解决了 JSON 格式无法表达链式调用的核心问题。
3. **LLM 对比**：
   - **deepseek-reasoner** 在 `--script` 模式下表现最好（97.1%/85.2%），但耗时最长
   - **google (gemini-2.5-flash)** 在默认模式下生成用例最多，准确率较高
   - **github (gpt-4o-mini)** 速度最快但覆盖率和准确率偏低
   - **deepseek-chat** 性价比最均衡
4. **需求文档的作用**：有需求文档时，准确率（通过率）普遍提升，但对覆盖率影响不大。

## 六、Prompt 设计总结

工具共使用了 **5 种 Prompt**，分别用于不同阶段：

| Prompt | 用途 | 输出格式 |
|--------|------|----------|
| 初始 prompt | 首次生成测试用例 | JSON 数组 |
| AST prompt | 含 AST 分析的首次生成 | JSON 数组 |
| 补充 prompt | 闭环补充（默认模式） | JSON 数组 |
| 脚本生成 prompt | `--script` 模式首次脚本生成 | 完整测试脚本代码 |
| 片段补充 prompt | `--script` 模式闭环补充 | 测试方法片段代码 |

详细的 Prompt 内容和设计说明见 `docs/prompt记录.md`。

## 七、工具使用方式

```bash
# 基本用法：只生成 JSON 测试用例
python main.py --source targets/convert_number_to_words.py --llm deepseek-chat

# --script 模式：生成测试脚本 + 覆盖率闭环
python main.py --source targets/index.js --llm deepseek-chat --script

# AST 辅助（仅 Python）
python main.py --source targets/convert_number_to_words.py --llm deepseek-chat --ast

# 附带需求文档
python main.py --source targets/convert_number_to_words.py --llm deepseek-chat --requirement targets/requirements_convert_number_to_words.md

# 多 LLM 对比
python main.py --source targets/convert_number_to_words.py --compare
```
