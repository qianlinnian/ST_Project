# Defense Script

## Slide 1 Title

Good morning, teachers. Our project is called White-box Test Case Generation Tool.

老师好，我们的项目名称是白盒测试用例生成工具。

This project explores how large language models can be integrated into a white-box testing workflow to improve test generation efficiency and coverage.

这个项目探索的是，如何将大语言模型整合进白盒测试流程中，以提升测试生成效率和覆盖率。

Our goal is not to let AI replace testing, but to use AI as an assistant for code analysis, test generation, and iterative refinement.

我们的目标不是让 AI 替代测试，而是让 AI 作为代码分析、测试生成和迭代补充的辅助工具。

Today, I will introduce our motivation, tool design, experimental results, engineering challenges, and final conclusions.

今天我将介绍我们的项目动机、工具设计、实验结果、工程挑战以及最终结论。

## Slide 2: 1.1 Manual White-box Testing

Before designing the tool, we first established a manual white-box testing baseline.

在设计工具之前，我们首先建立了一个人工白盒测试基线。

Our manual methodology contains three steps.

我们的人工方法主要包括三个步骤。

First, we construct the control flow graph, or CFG.

第一步是构建控制流图，也就是 CFG。

Second, we compute cyclomatic complexity.

第二步是计算圈复杂度。

Third, we derive independent paths and design test cases based on those paths.

第三步是根据独立路径设计测试用例。

We selected representative functions from both Python and JavaScript targets.

我们从 Python 和 JavaScript 两个目标程序中选取了有代表性的函数。

For Python, we used `max_value`, `convert_small_number`, and `convert_number`.

对于 Python，我们选取了 `max_value`、`convert_small_number` 和 `convert_number`。

For JavaScript, we selected `parseDate`, `parseLocale`, and `diff`.

对于 JavaScript，我们选取了 `parseDate`、`parseLocale` 和 `diff`。

This manual analysis is important, because it gives us a testing baseline and helps us compare AI-generated tests with human-designed tests.

这种人工分析非常重要，因为它为我们提供了测试基线，也让我们能够把 AI 生成的测试与人工设计的测试进行比较。

## Slide 3: 1.2 AI-based Test Generation Tool

Based on the manual testing workflow, we built an AI-based test generation tool.

基于人工测试流程，我们构建了一个基于 AI 的测试生成工具。

The tool architecture includes several stages.

这个工具的架构包含几个阶段。

First, the system reads and analyzes the target code.

首先，系统读取并分析目标代码。

Then it builds prompts and sends them to the LLM.

然后它构造提示词并发送给 LLM。

The generated test cases or scripts are executed automatically, and coverage is measured.

生成出的测试用例或脚本会被自动执行，并统计覆盖率。

If coverage is still incomplete, the tool enters a closed-loop refinement stage to generate supplementary tests.

如果覆盖率仍然不完整，工具会进入闭环补充阶段，生成额外测试。

We designed several prompt modes for different scenarios.

我们针对不同场景设计了几种提示模式。

These include general generation, AST-assisted mode for Python, script generation for complex cases, and supplement prompts for closed-loop refinement.

这些模式包括通用生成、面向 Python 的 AST 辅助模式、面向复杂场景的脚本生成模式，以及用于闭环补充的提示模式。

So the tool is not just a one-shot generator. It is a feedback-driven testing workflow.

所以这个工具并不是一次性生成器，而是一个由反馈驱动的测试工作流。

## Slide 4: 2. Experimental Results & Analysis

We evaluated the tool using several metrics.

我们使用多个指标来评估这个工具。

The first metric is coverage, including statement coverage and branch coverage.

第一个指标是覆盖率，包括语句覆盖率和分支覆盖率。

The second metric is accuracy, which mainly refers to pass rate and whether the generated tests are logically correct.

第二个指标是准确率，主要指通过率以及生成测试在逻辑上是否正确。

The third metric is test code quality, such as syntax correctness, framework compatibility, and logical validity.

第三个指标是测试代码质量，例如语法正确性、框架兼容性和逻辑有效性。

Our main findings are listed on the right.

我们的主要发现列在右侧。

First, closed-loop refinement helps improve coverage by generating tests for uncovered code.

第一，闭环补充可以通过为未覆盖代码生成测试来提高覆盖率。

Second, AST guidance improves test correctness because it provides structured branch information.

第二，AST 辅助能够提升测试正确性，因为它提供了结构化的分支信息。

Third, for complex JavaScript cases, direct script generation works better than JSON-based generation.

第三，对于复杂的 JavaScript 场景，直接生成脚本比基于 JSON 的生成效果更好。

From the representative results, we can see that Python performance is relatively stable, while JavaScript performance varies much more.

从代表性结果中可以看到，Python 的表现相对稳定，而 JavaScript 的波动要大得多。

This shows that structured code is easier for the tool, while complex chaining and prototype-related behavior remain challenging.

这说明结构化代码更容易被工具处理，而复杂链式调用和原型相关行为仍然具有挑战性。

## Slide 5: 3.1 JSON Limitation to Direct Script Generation

The first engineering challenge comes from JSON representation.

第一个工程挑战来自 JSON 表达方式本身。

For JavaScript libraries such as Dayjs, many operations involve chained calls like `dayjs().diff()`.

对于像 Dayjs 这样的 JavaScript 库，很多操作都涉及链式调用，比如 `dayjs().diff()`。

JSON is not expressive enough to describe this kind of behavior naturally.

JSON 并不足以自然地表达这类行为。

As a result, the generated tests often had framework misuse or structural errors.

结果就是，生成出的测试经常出现框架误用或结构错误。

To solve this problem, we introduced script mode.

为了解决这个问题，我们引入了脚本模式。

In this mode, the LLM directly generates executable test scripts instead of structured JSON test cases.

在这种模式下，LLM 不再生成结构化 JSON 用例，而是直接生成可执行的测试脚本。

During closed-loop refinement, the system only appends new fragments instead of rewriting the whole test file.

在闭环补充过程中，系统只追加新的片段，而不是重写整个测试文件。

This significantly improved JavaScript results.

这显著改善了 JavaScript 的测试效果。

## Slide 6: 3.2 Missing Syntax in AST to Analyzer Extension

The second challenge was incomplete AST analysis.

第二个挑战是 AST 分析不完整。

Our initial AST analyzer missed Python 3.10 `match-case` branches.

我们最初的 AST 分析器遗漏了 Python 3.10 的 `match-case` 分支。

This means some branch information was missing, which could reduce the quality of prompt guidance and test generation.

这意味着部分分支信息没有被提取出来，从而会影响提示词引导和测试生成质量。

After identifying this problem, we extended the analyzer to recognize `match-case` branches.

在发现这个问题之后，我们扩展了分析器，使它能够识别 `match-case` 分支。

This experience also taught us that static analyzers must evolve continuously with programming language syntax.

这段经历也让我们认识到，静态分析器必须随着编程语言语法不断演进。

## Slide 7: 3.3 Context Compression to Invalid Test Cases

The third challenge was context compression.

第三个挑战是上下文压缩。

In the supplementation stage, if we only provided a small local code snippet, the model sometimes generated invalid inputs.

在补充测试阶段，如果我们只提供一小段局部代码，模型有时会生成无效输入。

A typical example is `convert_small_number(100)`.

一个典型例子是 `convert_small_number(100)`。

Although this input looks related to the target line, it is blocked by entry validation before reaching the intended branch.

虽然这个输入看起来和目标代码有关，但它在到达目标分支之前就被入口校验拦截了。

This showed that local code alone may not provide enough logical context.

这说明仅靠局部代码并不能提供足够的逻辑上下文。

So a better strategy is to provide richer function context and make input constraints more explicit in prompts.

因此，更合理的策略是提供更丰富的函数上下文，并在提示词中更明确地表达输入约束。

This is a reminder that reducing prompt length should not sacrifice logical integrity.

这也提醒我们，减少提示词长度不能以牺牲逻辑完整性为代价。

## Slide 8: 3.4 LLM Common Sense vs Code Reality

The fourth challenge is the conflict between LLM common sense and actual code behavior.

第四个挑战是 LLM 常识与实际代码行为之间的冲突。

In the Dayjs project, the model sometimes assumed intuitive rounding behavior.

在 Dayjs 项目中，模型有时会假设符合直觉的四舍五入行为。

However, the actual implementation follows library-specific flooring logic.

但实际实现遵循的是库特定的向下取整逻辑。

This caused incorrect assertions.

这会导致错误断言。

On the right, we show one false assertion and one more appropriate assertion.

在右侧，我们展示了一个错误断言和一个更合适的断言。

This problem tells us that generated assertions should be guided by code facts, not just by semantic intuition.

这个问题告诉我们，生成的断言必须由代码事实来引导，而不能只依赖语义直觉。

To mitigate this issue, we need to describe library behavior more explicitly and use assertion styles such as `toBeCloseTo()` when floating-point outputs are involved.

为缓解这个问题，我们需要更明确地描述库行为，并在涉及浮点输出时使用 `toBeCloseTo()` 这类更合适的断言方式。

## Slide 9: 3.5 Output Instability to Fault-tolerant Engineering

The fifth challenge was unstable output format.

第五个挑战是输出格式不稳定。

LLM outputs sometimes contained extra explanations, malformed JSON, or syntax errors.

LLM 的输出有时会包含额外解释、错误格式的 JSON，或者语法错误。

Any of these could break the testing pipeline.

这些问题中的任何一种都可能破坏测试流程。

To make the system robust, we designed fault-tolerant engineering mechanisms.

为了让系统更稳健，我们设计了容错工程机制。

The first mechanism is multi-layer parsing.

第一个机制是多层解析。

The system tries direct loading first, then regex extraction, and finally line-level repair.

系统会先尝试直接加载，再尝试正则提取，最后进行逐行修复。

The second mechanism is version tracking and rollback.

第二个机制是版本跟踪和回滚。

If a later generated version is worse, the system can fall back to the best historical version.

如果后续生成的版本效果更差，系统可以回退到历史最佳版本。

This is one of the key engineering reasons why the tool remains usable even when LLM outputs are imperfect.

这也是即使 LLM 输出并不完美，整个工具仍然可用的重要工程原因之一。

## Slide 10: 4. Conclusion & Future Work

Finally, I would like to summarize our conclusions and contributions.

最后，我想总结一下我们的结论和贡献。

First, AI-generated testing is comparable to manual testing for structured Python code.

第一，对于结构化的 Python 代码，AI 生成测试的表现可以接近人工测试。

Second, the approach is still immature for complex JavaScript scenarios such as chaining and prototype-related behavior.

第二，对于链式调用和原型相关等复杂 JavaScript 场景，这种方法仍然不够成熟。

Third, the key insight of this project is that robust engineering systems matter more than perfect one-shot LLM outputs.

第三，这个项目最重要的认识是，稳健的工程系统比追求一次性完美的 LLM 输出更重要。

In terms of practical contributions, we built a complete LLM-based white-box testing tool.

在实际贡献方面，我们构建了一个完整的基于 LLM 的白盒测试工具。

It achieved full pipeline automation from code analysis to test generation and closed-loop refinement.

它实现了从代码分析到测试生成，再到闭环补充的全流程自动化。

Its key innovations include AST-integrated analysis, adaptive output modes, multi-LLM comparison, and a robust rollback mechanism.

它的关键创新包括 AST 集成分析、自适应输出模式、多 LLM 对比以及稳健的回滚机制。

For future work, we hope to enhance AST support, combine LLMs with symbolic execution and fuzzing, and further optimize prompt design.

在未来工作方面，我们希望增强 AST 支持，将 LLM 与符号执行和模糊测试结合，并继续优化提示词设计。

Overall, our project shows that AI can be useful in software testing, but only when it is integrated into a disciplined white-box testing workflow.

总体来说，我们的项目表明，AI 在软件测试中是有价值的，但前提是它必须被整合进一个有方法约束的白盒测试流程中。
