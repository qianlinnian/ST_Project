# Defense Script

## Slide 1 Title

Good morning, teachers. Our project is not simply an LLM tool demo, but an AI-assisted white-box testing workflow.

老师好，我们的项目并不只是一个 LLM 工具演示，而是一个 AI 辅助的白盒测试工作流。

The focus of this defense is on test requirements analysis, test design, and team-AI interaction during the testing process.

这次答辩的重点是测试需求分析、测试设计，以及测试过程中团队与 AI 的交互。

In our workflow, AI is responsible for generating and supplementing candidate tests, while the team is still responsible for defining coverage goals, checking test validity, and revising the final test plan.

在我们的工作流中，AI 负责生成和补充候选测试，而团队仍然负责确定覆盖目标、检查测试有效性，并修订最终测试方案。

So the key contribution is not replacing testing with AI, but integrating AI into a disciplined white-box testing process.

所以项目的关键贡献，不是用 AI 替代测试，而是把 AI 整合进一个有方法约束的白盒测试流程。

Today I will explain our requirements analysis, our test design, how prompts were used, how generated tests were analyzed and verified, and how the final plan was revised.

今天我将说明我们的需求分析、测试设计、提示词使用方式、生成测试的分析与验证方法，以及最终测试计划是如何修订形成的。

## Slide 2: 1.1 Manual White-box Testing

This slide focuses on test requirements analysis and manual baseline design.

这一页重点讲测试需求分析和人工基线设计。

Before using AI, we first identified the testing targets and coverage requirements manually.

在使用 AI 之前，我们先人工识别了测试目标和覆盖需求。

For Python, we chose `max_value`, `convert_small_number`, and `convert_number` because they represent different levels of branching complexity, input constraints, and path composition.

对于 Python，我们选择了 `max_value`、`convert_small_number` 和 `convert_number`，因为它们分别代表了不同层次的分支复杂度、输入约束和路径组合。

For JavaScript, we chose `parseDate`, `parseLocale`, and `diff` because they represent parsing behavior, locale-related logic, and the most complex date-difference calculation path.

对于 JavaScript，我们选择了 `parseDate`、`parseLocale` 和 `diff`，因为它们分别代表了解析逻辑、locale 相关逻辑，以及最复杂的日期差值计算路径。

Our manual white-box method followed three steps: constructing CFGs, computing cyclomatic complexity, and deriving independent paths for test case design.

我们的人工白盒方法遵循三个步骤：构建控制流图、计算圈复杂度，以及根据独立路径设计测试用例。

This manual process was important because it gave us the initial test plan, the branch coverage targets, and a baseline for judging whether AI-generated tests were meaningful.

这个人工过程很重要，因为它给了我们初始测试计划、分支覆盖目标，以及判断 AI 生成测试是否有意义的基线。

In other words, the team first decided what should be tested, and only then asked AI to help with generation.

换句话说，是团队先决定“应该测什么”，然后才让 AI 帮助生成测试。

## Slide 3: 1.2 AI-based Test Generation Tool

This slide focuses on how the team translated the test plan into an AI-assisted workflow.

这一页重点讲团队如何把测试计划转化为一个 AI 辅助工作流。

After defining the testing requirements, we built a tool that connects code analysis, prompt construction, test generation, execution, coverage measurement, and closed-loop refinement.

在确定测试需求之后，我们构建了一个工具，把代码分析、提示词构造、测试生成、测试执行、覆盖率测量和闭环补充连接起来。

The prompt modes correspond to different testing needs.

这些 prompt 模式分别对应不同的测试需求。

General generation is used for standard targets, AST-assisted mode is used when structured branch information is important, script mode is used when JSON cannot express the target behavior well, and supplement prompts are used to cover the remaining uncovered code.

通用生成用于标准目标，AST 辅助模式用于结构化分支信息很重要的场景，script 模式用于 JSON 难以准确表达目标行为的场景，而 supplement prompt 用于补足剩余未覆盖代码。

So prompt design in our project was not random prompt writing; it was derived from the test design itself.

所以我们项目中的 prompt 设计并不是随意写提示词，而是从测试设计本身推导出来的。

The team decided what information AI should receive, such as source code, requirements, branch information, and uncovered regions, and then adjusted the interaction mode according to the code style.

团队决定了 AI 应该接收哪些信息，比如源码、需求文档、分支信息和未覆盖区域，然后再根据代码风格调整交互方式。

This is the core idea of team-AI interaction in our tool.

这就是我们工具里团队与 AI 交互的核心思路。

## Slide 4: 2. Experimental Results & Analysis

This slide focuses on generated test case analysis, verification, and revision of the test plan.

这一页重点讲生成测试的分析、验证，以及测试计划的修订。

We evaluated the generated results using three dimensions: coverage, accuracy, and test code quality.

我们从三个维度评估生成结果：覆盖率、准确率和测试代码质量。

The important point is that we did not treat higher coverage as automatic success.

重要的一点是，我们并没有把更高的覆盖率直接当成成功。

Instead, the team checked whether generated tests really passed, whether the assertions were logically correct, and whether the scripts matched the target framework.

相反，团队会检查生成测试是否真的通过、断言在逻辑上是否正确，以及脚本是否匹配目标测试框架。

The experimental results gave us three design conclusions.

实验结果让我们得到三个设计结论。

First, closed-loop refinement can find uncovered code and improve coverage, but the final validity still needs human review.

第一，闭环补充可以发现未覆盖代码并提升覆盖率，但最终有效性仍然需要人工审查。

Second, AST guidance improves correctness because structured branch information helps AI understand what paths should be tested.

第二，AST 辅助能提升正确性，因为结构化分支信息能帮助 AI 理解“哪些路径应该被测试”。

Third, direct script generation works better than JSON for complex JavaScript, which means test representation itself is part of test design.

第三，对于复杂 JavaScript，直接生成脚本比 JSON 更有效，这说明测试表达方式本身也是测试设计的一部分。

So this page is not only about reporting numbers, but also about how results fed back into our testing strategy.

所以这一页不仅是在报数字，更是在说明实验结果如何反过来修正我们的测试策略。

## Slide 5: 3.1 JSON Limitation to Direct Script Generation

This slide is our first concrete example of team-AI interaction and plan revision.

这一页是团队–AI 交互和计划修订的第一个具体例子。

Our original testing problem was that JSON-based test cases could not represent chained JavaScript calls such as `dayjs().diff()` clearly enough.

我们最初遇到的测试问题是，基于 JSON 的测试用例无法清晰表达像 `dayjs().diff()` 这样的链式 JavaScript 调用。

As a result, the AI often generated structurally awkward cases or framework-misaligned outputs.

结果就是，AI 经常生成结构别扭的用例，或者与测试框架不匹配的输出。

The team verified this by checking the generated cases and observing that the representation itself had become the bottleneck.

团队通过检查生成用例发现，问题的瓶颈已经不是单个断言，而是整个测试表达方式本身。

So we revised the plan: instead of asking AI for JSON test cases, we asked it to generate executable scripts directly.

所以我们修订了测试计划：不再要求 AI 输出 JSON 用例，而是直接让它生成可执行脚本。

This is a good example showing that when the abstraction is wrong, the interaction protocol with AI must also be changed.

这就是一个很典型的例子，说明当抽象方式不合适时，与 AI 的交互协议也必须跟着改变。

## Slide 6: 3.2 Missing Syntax in AST to Analyzer Extension

This slide shows how human analysis corrected missing structural information before sending it to AI.

这一页说明团队如何先修正结构信息缺失，再把更准确的信息交给 AI。

The original testing problem was that our early AST analyzer did not recognize Python 3.10 `match-case` branches.

最初的问题是，我们早期的 AST 分析器无法识别 Python 3.10 的 `match-case` 分支。

That meant the branch information used in prompt construction was incomplete.

这意味着被送进 prompt 的分支信息本身就是不完整的。

We discovered this by comparing real source code structure with the extracted branch list during analysis of the Python target.

我们是在分析 Python 目标程序时，通过对照真实源码结构和提取出来的 branch list 发现这个问题的。

After verification, we revised the analysis module so that `match-case` branches could be recognized and included in the structured information.

在验证之后，我们修订了分析模块，使 `match-case` 分支能够被识别并加入结构化信息中。

This case shows that AI generation quality depends heavily on the correctness of the human-designed analysis layer.

这个案例说明，AI 生成质量很大程度上依赖于人工设计的分析层是否正确。

## Slide 7: 3.3 Context Compression to Invalid Test Cases

This slide focuses on how we analyzed invalid generated cases and revised the prompt strategy.

这一页重点讲我们如何分析无效生成结果，并修订 prompt 策略。

The original problem was that during supplementation, we sometimes gave AI only a narrow local code snippet in order to reduce prompt length.

最初的问题是，在补充测试阶段，我们有时为了缩短 prompt，只给 AI 一小段局部代码。

The generated result looked relevant, but some inputs were actually invalid, such as `convert_small_number(100)`, which was blocked by entry validation before reaching the target branch.

生成结果表面上看起来有关联，但有些输入实际上是无效的，例如 `convert_small_number(100)`，它在到达目标分支之前就被入口校验拦截了。

The team verified this by tracing the function preconditions and the dependency chain inside the target function.

团队是通过追踪函数前置条件以及目标函数内部的依赖链来验证这个问题的。

This led us to revise the interaction strategy: instead of only local snippets, we should provide richer function context and make input constraints more explicit in prompts.

这促使我们修订了交互策略：不能只给局部片段，而应该提供更丰富的函数上下文，并在 prompt 中更明确地表达输入约束。

So this case demonstrates that prompt design must preserve the logical integrity needed by the test plan.

所以这个案例说明，prompt 设计必须保留测试计划所需的逻辑完整性。

## Slide 8: 3.4 LLM Common Sense vs Code Reality

This slide shows how generated test cases were manually analyzed and corrected.

这一页说明生成测试是如何被人工分析和修正的。

The original problem was that AI sometimes produced assertions based on common sense rather than actual code semantics.

最初的问题是，AI 有时会根据“常识”生成断言，而不是根据真实代码语义生成断言。

In the Dayjs-related experiments, the model sometimes assumed intuitive rounding behavior, while the actual implementation followed library-specific flooring logic.

在 Dayjs 相关实验中，模型有时会假设符合直觉的四舍五入行为，而实际实现遵循的是库特定的向下取整逻辑。

We discovered this by comparing generated assertions with actual runtime behavior and by checking the implementation details of the library.

我们是通过对照生成断言与实际运行结果，并检查库的实现细节，发现这个问题的。

After this verification, we revised the testing strategy by describing library behavior more explicitly and using more appropriate floating-point assertions such as `toBeCloseTo()` when needed.

在完成验证后，我们通过更明确地描述库行为，并在需要时使用 `toBeCloseTo()` 这类更合适的浮点断言，来修订测试策略。

This case is important because it shows that human verification is essential even when AI generates high-coverage test cases.

这个案例很重要，因为它说明即使 AI 生成了高覆盖率测试，人工验证仍然是必不可少的。

## Slide 9: 3.5 Output Instability to Fault-tolerant Engineering

This slide focuses on how we verified unstable outputs and revised the overall workflow to make testing reliable.

这一页重点讲我们如何验证输出不稳定问题，并修订整体工作流，使测试流程更可靠。

The original problem was that AI outputs sometimes contained extra text, malformed JSON, or syntax errors, which could break the whole pipeline.

最初的问题是，AI 输出有时会包含额外文字、格式错误的 JSON，或者语法错误，从而破坏整个流程。

The team verified this problem by inspecting failed outputs and identifying repeated failure patterns rather than treating them as isolated accidents.

团队是通过检查失败输出，并识别重复出现的失败模式来验证这个问题的，而不是把它们看成偶发错误。

Based on this analysis, we revised the workflow itself.

基于这一分析，我们修订了整个工作流。

We added multi-layer parsing, syntax validation, and rollback to the best historical version, so that imperfect AI output would not directly destroy the final result.

我们加入了多层解析、语法校验和回滚到历史最佳版本的机制，使不完美的 AI 输出不会直接破坏最终结果。

So the final lesson here is that in AI-assisted testing, engineering fault tolerance is part of the test design, not just an implementation detail.

所以这一页最后想说明的是，在 AI 辅助测试中，工程容错本身就是测试设计的一部分，而不只是实现细节。

## Slide 10: 4. Conclusion & Future Work

This final content summarizes the testing conclusions, the practical workflow we formed, and future directions.

这一页总结的是测试结论、我们最终形成的实践流程，以及未来方向。

Our first conclusion is that for structured Python code, AI can improve efficiency on top of a human-designed white-box testing baseline.

第一个结论是，对于结构化的 Python 代码，AI 可以在人工设计的白盒测试基线之上提升效率。

Our second conclusion is that for complex JavaScript, AI still needs stronger human review and more suitable output modes.

第二个结论是，对于复杂 JavaScript，AI 仍然需要更强的人工审查和更合适的输出模式。

Our third conclusion is that higher coverage does not automatically mean sufficient testing; assertion correctness and human verification still matter.

第三个结论是，更高的覆盖率并不自动意味着测试已经充分；断言正确性和人工验证仍然非常重要。

In terms of contribution, we did not only build a tool, but also formed a complete process: requirements analysis, test design, AI generation, human verification, and revision of the test plan.

从贡献上说，我们不仅实现了一个工具，也形成了一套完整流程：需求分析、测试设计、AI 生成、人工验证，以及测试计划修订。

For future work, we want to strengthen AST analysis, combine LLMs with symbolic execution and fuzzing, and improve test validity verification further.

对于未来工作，我们希望增强 AST 分析能力，将 LLM 与符号执行和模糊测试结合，并进一步提高测试有效性验证能力。

So the final message of this project is that AI is valuable when it supports testing methodology, rather than replacing it.

所以这个项目最后想表达的是，AI 的价值在于支持测试方法，而不是替代测试方法。
