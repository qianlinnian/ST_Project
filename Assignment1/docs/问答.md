# Defense Q&A

## Q1. Is this a testing project or a tool project?

**English**

It is a testing-oriented tool project. The core goal is still software testing, especially white-box testing. We built the tool to support requirements analysis, test generation, coverage feedback, and iterative refinement, rather than to replace testing methodology itself.

**中文**

这是一个面向测试的工具项目。核心目标仍然是软件测试，尤其是白盒测试。我们构建这个工具，是为了支持需求分析、测试生成、覆盖率反馈和迭代补充，而不是为了替代测试方法本身。

## Q2. What exactly did the team do, and what did AI do?

**English**

The team defined the testing targets, analyzed code structure, identified coverage goals, designed the initial testing plan, checked generated tests, and revised the final plan. AI was used to generate candidate test cases or scripts and to help supplement uncovered code.

**中文**

团队负责确定测试目标、分析代码结构、识别覆盖目标、设计初始测试计划、检查生成测试，并修订最终方案。AI 主要用于生成候选测试用例或测试脚本，并帮助补充未覆盖代码。

## Q3. How does your project satisfy the course requirement about test requirements analysis?

**English**

We did not directly ask AI to test the whole program blindly. We first selected representative functions, analyzed their control flow and complexity, and identified what kinds of branches, paths, and constraints should be covered. That was our test requirements analysis, and it determined what information we later provided to AI.

**中文**

我们并不是直接把整个程序丢给 AI 盲目测试。我们先选择了有代表性的函数，分析它们的控制流和复杂度，并识别出需要覆盖哪些分支、路径和约束。这部分就是我们的测试需求分析，它决定了后续给 AI 提供什么信息。

## Q4. How does your project satisfy the course requirement about test design?

**English**

Our test design started from manual white-box analysis. We constructed CFGs, computed cyclomatic complexity, derived independent paths, and then used those results to define the test baseline. After that, AI-assisted generation was used as an extension mechanism, not as a substitute for the original design.

**中文**

我们的测试设计从人工白盒分析开始。我们构建了控制流图，计算了圈复杂度，推导了独立路径，并用这些结果确定测试基线。之后，AI 辅助生成只是一个扩展机制，而不是对原始测试设计的替代。

## Q5. Where is the black-box part in your project?

**English**

The main focus of the project is white-box testing, but black-box thinking still appears in input constraints, requirement documents, valid input ranges, and boundary-aware prompts. In other words, we did not rely only on internal structure; we also considered external input rules and expected behavior.

**中文**

项目的主线是白盒测试，但黑盒测试的思路也体现在输入约束、需求文档、有效输入范围以及边界相关的提示词设计中。换句话说，我们并不只依赖代码内部结构，也考虑了外部输入规则和预期行为。

## Q6. Where is the experience-based testing part in your project?

**English**

Experience-based testing appears when we inspect common failure patterns and risky cases. For example, in the JavaScript target, we paid extra attention to chained calls, floating-point assertions, library-specific flooring behavior, and invalid inputs. These are not only structural issues but also experience-driven risk points.

**中文**

经验测试体现在我们分析常见失败模式和高风险场景时。例如，在 JavaScript 目标程序中，我们特别关注链式调用、浮点断言、库特定的向下取整行为以及无效输入。这些问题不只是结构问题，也属于经验驱动的风险点。

## Q7. Why did you choose those functions as test subjects?

**English**

We chose functions that represent different testing difficulties. In Python, the selected functions cover branch-heavy logic, input validation, and multi-step conversion. In JavaScript, the selected functions cover parsing, locale-related logic, and the most complex date-difference computation. They are representative rather than arbitrary.

**中文**

我们选择这些函数，是因为它们分别代表了不同类型的测试难点。在 Python 中，它们覆盖了高分支逻辑、输入校验和多步骤转换；在 JavaScript 中，它们覆盖了解析、locale 相关逻辑以及最复杂的日期差值计算。这些选择是有代表性的，而不是随意选的。

## Q8. What does your coverage analysis mean in practice?

**English**

Coverage analysis tells us how much of the code structure was actually executed, such as statements and branches. But in practice, we do not treat coverage as the only success criterion. We also check whether the generated assertions are correct, whether the tests pass, and whether the inputs are valid and meaningful.

**中文**

覆盖率分析告诉我们代码结构中有多少真正被执行了，比如语句和分支。但在实践中，我们并不把覆盖率当作唯一成功标准。我们还会检查生成断言是否正确、测试是否通过，以及输入是否有效且有意义。

## Q9. Why can AI have higher coverage than manual testing but still not be enough?

**English**

Because higher coverage only means that more code was executed. It does not automatically mean that the assertions are correct or that the test inputs are valid. AI can reach more branches but still misunderstand the implementation or generate weak assertions. That is why human verification remains necessary.

**中文**

因为更高的覆盖率只意味着更多代码被执行了，并不自动代表断言是正确的，或者测试输入是有效的。AI 可以覆盖更多分支，但仍然可能误解实现逻辑，或者生成较弱的断言。这就是为什么人工验证仍然必要。

## Q10. How did you verify whether AI-generated tests were good or bad?

**English**

We used several verification methods. We checked pass rate, statement and branch coverage, syntax correctness, framework compatibility, and whether the assertions matched actual code behavior. When a generated test looked suspicious, we manually traced the code logic and compared expected results with real execution behavior.

**中文**

我们用了多种验证方法。我们检查测试通过率、语句和分支覆盖率、语法正确性、框架兼容性，以及断言是否与实际代码行为一致。当某个生成测试看起来可疑时，我们会人工追踪代码逻辑，并比较预期结果和真实执行行为。

## Q11. How did prompts help your testing process?

**English**

Prompts were used as a way to translate the test plan into instructions that AI could follow. We used different prompt modes depending on the testing task, such as general generation, AST-assisted generation, script generation, and supplement prompts. So prompt design in our project was part of test design, not an isolated writing task.

**中文**

Prompt 的作用，是把测试计划转化为 AI 可以执行的指令。我们根据不同测试任务使用了不同的 prompt 模式，比如通用生成、AST 辅助生成、脚本生成和补充生成。所以在我们的项目中，prompt 设计是测试设计的一部分，而不是孤立的写提示词工作。

## Q12. What is the meaning of AST assistance in your project?

**English**

AST assistance provides structured code information to the model, such as functions, branches, and paths. This helps AI understand what should be tested in a more disciplined way. It does not replace source code, but it improves the quality of structural guidance.

**中文**

AST 辅助的作用，是给模型提供结构化代码信息，比如函数、分支和路径。这能帮助 AI 以更有约束的方式理解“应该测什么”。它并不替代源码本身，但会提升结构化引导的质量。

## Q13. Why did you add script mode?

**English**

We added script mode because JSON-based test representation was not expressive enough for complex JavaScript chaining behavior. Script mode allowed the model to generate executable test scripts directly, which better matched the actual testing framework and the code style of the target project.

**中文**

我们加入 script 模式，是因为基于 JSON 的测试表示方式不足以表达复杂的 JavaScript 链式调用行为。script 模式允许模型直接生成可执行测试脚本，这样更符合真实测试框架，也更符合目标项目的代码风格。

## Q14. Why did some AI-generated test cases fail?

**English**

The failures usually came from three sources: incomplete structural understanding, insufficient context, or incorrect assumptions about library behavior. For example, AI could generate invalid inputs, misunderstand floating-point and flooring behavior, or produce syntax and framework errors.

**中文**

这些失败通常来自三个来源：结构理解不完整、上下文不足，或者对库行为做出了错误假设。例如，AI 可能生成无效输入、误解浮点和向下取整行为，或者产生语法和框架错误。

## Q15. How did the team revise the test plan during the project?

**English**

We revised the plan whenever generated results exposed a systematic issue. For example, when JSON was not expressive enough, we switched to script mode. When AST information was incomplete, we extended the analyzer. When local snippets caused invalid inputs, we adjusted context strategy. So the final version was formed through repeated analysis, verification, and revision.

**中文**

每当生成结果暴露出系统性问题时，我们就会修订测试计划。例如，当 JSON 表达能力不足时，我们切换到 script 模式；当 AST 信息不完整时，我们扩展分析器；当局部代码片段导致无效输入时，我们调整上下文策略。所以最终版本是通过反复分析、验证和修订形成的。

## Q16. Why do you say this is team–AI interaction rather than just AI generation?

**English**

Because AI did not work independently. The team decided the targets, designed prompts, checked outputs, identified errors, revised the workflow, and verified the final tests. AI generated content, but the process was guided and corrected by the team throughout.

**中文**

因为 AI 并不是独立工作的。团队决定测试目标、设计 prompt、检查输出、识别错误、修订工作流，并验证最终测试。AI 负责生成内容，但整个过程始终由团队进行引导和修正。

## Q17. What are the limitations of your current project?

**English**

The main limitations are that AI still struggles with complex JavaScript behavior, some syntax structures require continuous analyzer updates, and high coverage does not guarantee high-quality assertions. In addition, generated results still require manual review and cannot yet fully replace disciplined testing work.

**中文**

当前项目的主要局限是：AI 仍然难以稳定处理复杂 JavaScript 行为，一些语法结构需要持续更新分析器，而且高覆盖率并不保证高质量断言。此外，生成结果仍然需要人工审查，还不能完全替代有方法约束的测试工作。

## Q18. What is the main contribution of your project?

**English**

The main contribution is that we formed a complete testing workflow: requirements analysis, test design, AI generation, human verification, and revision of the final test plan. So our contribution is not only a tool, but also a practical process for integrating AI into white-box testing.

**中文**

项目的主要贡献是，我们形成了一套完整的测试流程：需求分析、测试设计、AI 生成、人工验证，以及最终测试计划的修订。所以我们的贡献不只是一个工具，也是一种将 AI 融入白盒测试的实践流程。

## Q19. If you continue this project, what would you improve next?

**English**

We would improve AST support for more syntax structures, strengthen test validity verification, and combine LLM-based generation with more traditional techniques such as symbolic execution and fuzzing. These directions would improve both coverage quality and reliability.

**中文**

如果继续做这个项目，我们会扩展更多语法结构的 AST 支持，增强测试有效性验证，并把基于 LLM 的生成与符号执行、模糊测试等更传统的技术结合起来。这些方向会同时提升覆盖质量和可靠性。

## Q20. Why is human review still necessary in your project?

**English**

Human review is necessary because testing is not only about producing outputs. It is also about deciding what should be covered, judging whether assertions are meaningful, and revising the plan when the generated results are misleading. AI can accelerate the process, but it cannot yet replace responsible testing judgment.

**中文**

人工审查仍然必要，因为测试并不只是“生成结果”这么简单。它还包括决定应该覆盖什么、判断断言是否有意义，以及在生成结果具有误导性时修订测试计划。AI 可以加速这个过程，但还不能替代负责任的测试判断。
