                         实验分析报告
本次实验围绕 convert_number_to_words（Python 纯函数项目）和 Dayjs（JavaScript

链式调用日期库项目）两个被测对象，基于 DeepSeek Chat、DeepSeek Reasoner、Google Gemini-2.5-Flash、GitHub GPT-4o-mini 四款大语言模型（LLM），开展了多维度白盒测试用例自动生成实验。实验设置了无需求文档/有需求文档、无 AST 分析/有 AST 分析、默认JSON 用例/--script 脚本生成三种变量，从准确性、覆盖率、泛化性三个核心维度对比了不同 LLM 的表现，并与人工测试用例进行了对照。实验结果表明，LLM 在白盒测试用例生成中具备较高的可行性，AST 分析和需求文档的引入可显著提升用例质量，--script 模式则解决了 JavaScript 链式调用场景下 JSON 格式表达能力不足的问题；同时不同 LLM 在性能、准确率、覆盖率上存在明显差异，DeepSeek Reasoner 综合表现最优，DeepSeek Chat 性价比均衡，GPT-4o-mini 速度最快但效果偏弱，Gemini-2.5-Flash 生成用例数量多但准确率波动大。人工测试虽准确率高，但存在主观直觉偏差，且在测试效率和复杂路径覆盖上弱于 AI 工具。

一、准确性分析
测试用例的准确性以用例通过率为核心指标，即运行通过的测试用例数占总用例数的比例，同时结合用例有效性（是否符合被测函数的业务逻辑和输入输出规则）进行综合评判，
本次实验同时分析了人工测试用例和不同条件下 AI 生成用例的准确性表现。
1. 人工测试用例的准确性特征
人工测试用例在逻辑贴合度上具备天然优势，但易受主观直觉偏差影响，存在局部准确率陷阱：
Python 项目（convert_number_to_words）：人工设计的 17 个测试用例严格覆盖了 3个核心函数的所有独立路径，执行通过率 100%，无逻辑错误，体现了人工对代码路径和业务规则的精准把控。
JavaScript 项目（Dayjs）：人工设计的 27 个测试用例通过率仅 29.63%，核心问题为两点：一是对 Dayjs.diff 方法的浮点数计算逻辑存在直觉偏差，使用严格相等断言（.toBe ()）判断非整数时间差，忽略了库中基于实际天数的换算规则；二是对 diff方法 float = false 时的取整逻辑误解，主观认为是四舍五入，而实际库中为向下截断/直接取整，导致期望值设置错误。
上述现象表明，人工测试的准确性高度依赖测试人员对源码实现细节的掌握，当测试人员仅
凭领域常识而非代码逻辑设计用例时，易出现“逻辑实现偏差”，导致用例失效。
2. AI 生成用例的准确性整体表现
不同 LLM 在两款被测项目中的准确率差异显著，且受被测项目类型、是否引入需求文档/AST分析、输出模式影响极大，核心数据如下：

Python 纯 函数 项 目：无 需求文 档、无 AST 分析 时，四 款 LLM 平均 准确率 约 72.5%（Gemini-2.5-Flash 90.70%、DeepSeek Chat 69.23%、GPT-4o-mini 70.83%、DeepSeekReasoner 59.26%）；引入需求文档后，平均准确率提升至 77.78%，其中 DeepSeek
Reasoner 提升最明显（59.26%→87.10%）；引入 AST 分析后，首轮即可实现 86.96% ~
95.45%的高准确率，DeepSeek Chat 达 94.44%、DeepSeek Reasoner 达 95.00%，且闭环
轮次从 3 轮降至 1 轮，避免了多轮补充导致的用例过拟合。

JavaScript 链式调用项目：默认 JSON 用例模式下，四款 LLM 准确率均为 0% ~ 0.36%，核心原因是 JSON 格式无法表达 Dayjs 的链式调用语法（如 dayjs ('2024-01-01').diff('2023-01-01', 'year')），生成的用例仅调用构造函数，未触发核心方法；开启--script 模式后，LLM 直接生成可运行的 Jest 测试脚本，准确率显著提升，DeepSeekChat 达 97.37%、DeepSeek Reasoner 达 62.22%，解决了语法表达的核心问题。

3. AI 准确性的关键影响因素
需求文档：为 LLM 提供了明确的业务规则和输入输出约束，减少了 LLM 对代码逻辑的“猜
测”，使准确率平均提升 5% ~ 10%，尤其对 DeepSeek Reasoner 这类推理型模型效果更显著。

AST 分析：通过 Python AST 模块精确提取代码的函数列表、分支点、复合条件和执行路径，注入 prompt 后为 LLM 提供了结构化的代码逻辑信息，避免了 LLM 因代码理解不完整导致的用例错误，使 Python 项目的准确率提升 20%以上，且大幅降低了闭环补充的轮次。

输出模式：JSON 模式适合纯函数、输入输出为简单类型的项目，--script 模式则适配链式调用、面向对象的项目，是提升 AI 在复杂语法场景下准确性的关键。

LLM 自身特性：推理型模型（DeepSeek Reasoner）对代码逻辑的理解更精准，生成用例的有效性更高；轻量型模型（GPT-4o-mini）速度快但易产生幻觉，用例错误率较高；生成型模型（Gemini-2.5-Flash）生成用例数量多，整体准确率较高，但存在部分低质量冗余用例。

二、覆盖率分析
覆盖率以语句覆盖率和分支覆盖率为核心指标，本次实验从人工测试 vs AI 测试、不同 LLM 的 AI vs AI 对比两个维度展开分析，同时分析了不同实验条件对 AI 覆盖率的影响。

1. 人工测试 vs AI 测试覆盖率对比
人工测试的覆盖率为人工基准值，AI 测试在多数场景下可达到甚至超越人工基准，且在复杂路径覆盖和多场景适配上表现更优：

Python 项目（convert_number_to_words）：人工测试实现 90%语句覆盖率、未统计分支覆盖率，未覆盖的代码行为第 51-52 行、201-205 行；四款 LLM 在无需求文档/无 AST分析时，均实现 91%语句覆盖率、92.3%分支覆盖率，略高于人工基准，且覆盖了人工未涉及的部分分支路径；引入 AST 分析和需求文档后，覆盖率保持 91%/92.3%的稳定水平，未出现下降。

JavaScript 项目（Dayjs）：人工测试未完成全量覆盖率统计，但从人工用例的设计逻辑来看，仅覆盖了 parseDate、parseLocale、diff 三个核心函数的基础路径；AI 在默认 JSON 模式下，语句覆盖率仅 26.5% ~ 58.4%、分支覆盖率 18.7% ~ 44.0%，仅覆盖构
造函数相关代码；开启--script 模式后，DeepSeek Reasoner 实现 97.1%语句覆盖率、85.2%分支覆盖率，DeepSeek Chat 实现 72.7%语句覆盖率、63.7%分支覆盖率，大幅覆盖了 Dayjs 的核心方法（如 format、startOf、isValid、add 等），远超人工测试的覆盖范围。

人工测试覆盖率的局限性在于测试人员的精力和思维盲区，人工难以在短时间内覆盖复杂项目的所有分支和路径，而 AI 可通过对代码的结构化分析和批量用例生成，实现更全面的覆盖；但 AI 在默认模式下存在格式表达能力不足的问题，导致覆盖率偏低，需通过模式优化解决。

2. 不同 LLM 的 AI vs AI 覆盖率对比
四款 LLM 在覆盖率上的表现与模型类型、被测项目类型、实验模式强相关，核心差异如下：

Python 项目：四款 LLM 的覆盖率高度一致，均为 91%语句覆盖率、92.3%分支覆盖率，无明显差异。 原因是 Python 项目为纯函数式设计，代码逻辑清晰，路径数有限，且JSON 模式可完美表达用例，各 LLM 均能通过基础的代码理解实现全量路径覆盖。

JavaScript 项目：开启--script 模式后，LLM 覆盖率差异显著，DeepSeek Reasoner（97.1%/85.2%）> DeepSeek Chat（72.7%/63.7%）> GPT-4o-mini（34.0%/20.9%），Gemini-2.5-Flash 未 参 与 --script 模 式 测 试 。 核 心 原 因 是 DeepSeek Reasoner 对JavaScript 链式调用语法和库的内部逻辑理解更精准，生成的用例能触发更多核心方法，而 GPT-4o-mini 作为轻量型模型，对复杂库的代码理解深度不足。

整体来看，推理型 LLM 在覆盖率上的表现优于生成型和轻量型 LLM，更适合复杂项目的白盒测试用例生成。

3. 实验条件对 AI 覆盖率的影响
AST 分析：对 Python 项目的覆盖率无明显提升（已达 91%/92.3%），但可使覆盖率首轮达标，避免了多轮闭环补充的耗时；

需求文档：对两款项目的覆盖率均无显著影响，仅提升用例准确率，说明覆盖率主要依赖 LLM 对代码结构的理解，而非业务规则；

--script 模式：是 JavaScript 项目覆盖率提升的核心因素，将语句覆盖率从不足 60%
提升至最高 97.1%，解决了 JSON 格式无法表达链式调用的核心痛点；

闭环补充机制：当首轮覆盖率未达目标时，通过将未覆盖的代码行/分支反馈给 LLM，补充生成专用用例，可使覆盖率平均提升 10% ~ 30%，是实现高覆盖率的重要保障。

三、泛化性分析
泛化性指 AI 生成的测试用例在不同类型被测项目、不同实验条件、不同代码逻辑场景
下的适配能力和可复用性，同时包括 AI 工具在闭环补充、跨语言场景下的表现，本次实验
从项目类型泛化、实验条件泛化、逻辑场景泛化三个维度分析 AI 的泛化能力。
1. 项目类型泛化：纯函数 vs 链式调用
LLM 在 Python 纯函数式项目中表现出极强的泛化能力，而在 JavaScript 链式调用项目中泛化能力弱，需依赖模式优化，核心差异如下：

Python 纯函数项目：无论是否引入需求文档/AST 分析，四款 LLM 均能生成符合语法和
逻辑的用例，JSON 格式可完美表达函数的输入输出，用例能直接触发所有核心代码路径，泛化能力优秀；

JavaScript 链式调用项目：默认 JSON 模式下，LLM 的泛化能力极差，生成的用例仅能调用构造函数，无法表达实例.方法()的链式语法，无法触发核心方法；开启--script模式后，LLM 可直接生成适配 Jest 框架的链式调用测试脚本，泛化能力显著提升，能覆盖 parseDate、parseLocale、diff、format、startOf 等多个核心函数，泛化能力良好。

这一现象表明，LLM 本身具备对不同代码逻辑的理解能力，但用例输出格式是制约其泛化性的关键因素，需根据项目类型设计自适应的输出模式。
2. 实验条件泛化：无需求/AST→有需求/AST
LLM 在引入需求文档和 AST 分析的条件下，表现出良好的泛化性，且未出现“过拟合”问题：引入需求文档：LLM 能将业务规则与代码逻辑结合，生成的用例既符合代码路径覆盖要求，又满足业务输入输出约束，在 Python 项目中，引入需求文档后的用例可直接复用
至不同 LLM，且准确率提升，无规则冲突；
引入 AST 分析：AST 提取的结构化代码信息为 LLM 提供了统一的逻辑参考，不同 LLM 基于 AST 信息生成的用例均能覆盖核心分支和路径，且在闭环补充中，未出现针对某一代码行的“过度拟合”用例，用例仍具备对同类逻辑的适配能力。

此外，在多轮闭环补充中，AI 的泛化能力略有下降，部分 LLM 会为覆盖特定未执行代码行，

生成不符合业务逻辑的极端用例，导致用例通过率降低，这是 LLM 生成测试用例的固有泛化性局限。

3. 逻辑场景泛化：基础路径 vs 复杂路径
LLM 在基础分支/路径（如简单 if 判断、单一层级循环）中泛化能力优秀，在复杂逻辑场景（如递归调用、复合条件、多分支 switch、时区/浮点数等特殊计算）中，泛化能力受模型类型影响较大：

递归调用（convert_number 的递归分支）：四款 LLM 均能生成触发递归的用例（如 345、12345），泛化能力优秀，且能覆盖递归中的嵌套分支；

复合条件（如 num > 0 or not word_groups）：LLM 能生成覆盖复合条件各子条件的用例，如输入 0 覆盖 not word_groups，输入正数覆盖 num > 0，泛化能力良好；

多分支 switch（Dayjs.diff 的 unit 判断）：推理型模型（DeepSeek Reasoner/DeepSeekChat）能生成覆盖所有 switch 分支（年、月、季度、周、日等）的用例，轻量型模型（GPT-4o-mini）仅能覆盖基础分支（天、小时），泛化能力一般；

特殊计算（Dayjs.diff 的浮点数/取整、时区偏移）：DeepSeek Reasoner 能生成适配库计算逻辑的用例，使用容错断言（.toBeCloseTo ()）判断浮点数，符合向下截断的取整规则，泛化能力优秀；其他模型则易出现与人工类似的直觉偏差，泛化能力一般。

四、实验总结
本次实验从准确性、覆盖率、泛化性三个维度验证了基于 LLM 的白盒测试用例自动生成

工具的可行性和有效性，核心结论如下：

1. AI 测试在覆盖率上可达到甚至超越人工测试，且在复杂项目的路径覆盖上具备显著优势；人工测试在基础路径的准确率上更高，但易受主观直觉偏差影响，且存在精力和思
维盲区。

2. 不同 LLM 的表现差异显著：DeepSeek Reasoner 在推理能力、覆盖率、准确率上综合表现最优，适合复杂项目；DeepSeek Chat 性价比均衡，准确率和覆盖率较高且耗时适中；Gemini-2.5-Flash 生成用例数量多，准确率较高，但存在冗余用例；GPT-4o-mini 速度最快，但覆盖率和准确率偏低，适合简单项目。

3.  AST 分析和需求文档是提升 AI 用例质量的关键：AST 分析使 Python 项目闭环轮次从 3轮降至 1 轮，准确率提升 20%以上；需求文档使 AI 用例准确率平均提升 5% ~ 10%，减少逻辑猜测。

4.  --script 模式是解决 JavaScript 链式调用场景下 JSON 格式表达能力不足的核心方案，将 Dayjs 项目的语句覆盖率从不足 60%提升至最高 97.1%。

5.  LLM 的泛化能力与项目类型和模型类型强相关：在 Python 纯函数项目中泛化能力优秀，
在 JavaScript 链式调用项目中需依赖--script 模式；推理型模型在复杂逻辑场景中的泛化能力显著优于轻量型模型。

6.  AI 测试存在固有局限性：闭环补充时易出现过拟合，生成不符合业务逻辑的极端用例；部分模型对复杂计算逻辑存在直觉偏差，需通过 prompt 优化和结构化信息注入改善。
